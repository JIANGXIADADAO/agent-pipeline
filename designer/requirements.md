## 三、RICE 优先级表

### 评分标准

| 维度 | 说明 | 评分范围 |
|------|------|---------|
| **Reach** | 每阶段影响的用户数 | 1-5 |
| **Impact** | 对 JTBD 完成度的贡献 | 1-5 |
| **Confidence** | 需求确定性 | 1-5（50%/75%/90%/95%/99%） |
| **Effort** | 工程复杂度 | 1-10（10 最易） |

## 四、Phase 1 MVP 范围（精确边界）

### In Scope

| 组件 | 说明 |
|------|------|
| Orchestrator | 单 Agent 路由 → 接收需求 → 分派给 Scout → 等待完成 → 输出结果 |
| Scout Agent | 搜索竞品/技术/方案 → 读网页提取信息 → 写结构化调研报告 |
| CLI 入口 | `agent-pipeline run "做一个 xxx 产品"` |
| 状态持久化 | 每次状态变更记录到 state.json，支持断点恢复 |
| 简单错误处理 | Agent 超时（默认 5 分钟）→ 返回"Scout 调研失败，请重试" |

### Out of Scope

| 组件 | 说明 | 移至 |
|------|------|------|
| 其他四个 Agent | Designer / Builder / Tester / Seller | Phase 2 |
| Web UI | Pipeline 可视化 | Phase 2 |
| 预索引 RAG | 内部门控 | Phase 3 |
| SSE 推送 | 实时状态更新 | Phase 2 |
| 多轮对话 | 追问需求细节 | Phase 4 |
| CHANGELOG 生成 | 使用 changelog-tool | Phase 3 |
| PR 创建 | GitHub API | Phase 3 |

## 五、验收标准

### Phase 1 验收标准

1. `agent-pipeline run "调研 AI 编码助手市场"` → Scout 输出结构化调研报告（markdown 文件）
2. 支持 `agent-pipeline run --resume` 从断点恢复
3. 所有错误场景（无 API key/网络超时/Agent 空输出）有明确的错误信息
4. 运行耗时 ≤ 10 分钟（含 LLM 调用）

### Phase 2 验收标准

1. 输入需求 → 5 Agent 依次运行 → 输出完整项目（代码 + 测试 + README + CHANGELOG）
2. Web UI 实时展示 Agent 状态：队列→运行中→完成/失败
3. 支持点击查看每个 Agent 的详细输出
4. 端到端运行 ≤ 30 分钟

### Phase 3 验收标准

1. 预索引门控有效：命中 index.md 的直接跳 RAG → 更优的结果
2. LLM-as-judge 评分覆盖：质量/安全/一致性
3. Agent 失败自动重试 1 次后人工标记
